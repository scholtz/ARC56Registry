ARC-56 program hash + ABI signature registry

Look up a deployed Algorand app's compiled TEAL by SHA-256 hash and get back a real ARC-56 spec you can use to decode its calls - or look up a raw 4-byte ARC-4 method selector and get back the human-readable method signature it came from.

Why this exists

A wallet about to send an application call usually only has the app ID. It can fetch the app's compiled approval and clear programs from the network, but without a spec it has no idea what methods exist, what arguments they take, or how to render the call to a user instead of a wall of opaque bytes.

This registry closes that gap: every ARC-56 spec indexed by scholtz/ARC56Registry has its compiled approval and clear programs hashed with SHA-256, and each hash maps to a durable URL for a matching spec.

Folder layout

RegistryPath
Approval program hashapproval-programs/<hash[:3]>/<hash>.txt
Clear-state program hashclear-programs/<hash[:3]>/<hash>.txt
ABI method selectorabi-signatures/<selector[:2]>/<selector>.txt

Splitting on the hash's first few hex characters keeps any one folder from holding thousands of files. Each program-hash .txt file contains exactly one line: a raw.githubusercontent.com URL pinned to the commit that last touched the matching ARC-56 spec, so it keeps resolving to the exact spec that produced this hash even after the source file is edited or replaced later. Each abi-signatures/ file instead contains the plain-text ABI method signature itself, e.g. add(uint64,uint64)uint128.

How to look up a hash

  1. Fetch the app's compiled programs, e.g. via algod's GET /v2/applications/{app-id}, and base64-decode params.approval-program and/or params.clear-state-program.
  2. Compute sha256(program_bytes), hex-encoded, lowercase.
  3. Fetch this hash's file from this site - /approval-programs/<hash[:3]>/<hash>.txt or /clear-programs/<hash[:3]>/<hash>.txt - using a URL relative to wherever this page is hosted. A 404 just means this registry hasn't indexed a spec producing that hash yet, not an error.
  4. If found, its content is a URL to the matching ARC-56 JSON spec. Fetch that and use it to decode the app's methods, argument types, and state layout.

Example: program hash to spec

import base64, hashlib, urllib.request, json

app_id = 123456789
info = json.load(urllib.request.urlopen(f"https://mainnet-api.algonode.cloud/v2/applications/{app_id}"))
approval = base64.b64decode(info["params"]["approval-program"])
digest = hashlib.sha256(approval).hexdigest()

lookup_url = f"https://scholtz.github.io/ARC56Registry/approval-programs/{digest[:3]}/{digest}.txt"
try:
    spec_url = urllib.request.urlopen(lookup_url).read().decode().strip()
    spec = json.load(urllib.request.urlopen(spec_url))
    print("Found spec for", spec["name"])
except urllib.error.HTTPError:
    print("No ARC-56 spec indexed for this program yet")

How to look up a method selector

Every ARC-4/ARC-56 application call's first argument is a 4-byte method selector: the first 4 bytes of SHA-512/256 (not SHA-256) over the method's ABI signature string, name(argtype,argtype,...)returntype. If you've extracted that selector from a transaction but don't have the app's ARC-56 spec, this registry can still resolve it to a human-readable signature:

  1. Hex-encode the 4 selector bytes, lowercase, e.g. 8aa3b61f.
  2. Fetch /abi-signatures/<selector[:2]>/<selector>.txt - relative to wherever this page is hosted. A 404 just means this registry hasn't indexed a method with that selector yet, not an error.
  3. If found, its content is the plain-text ABI signature, e.g. add(uint64,uint64)uint128.

Example: method selector to signature

import urllib.request

selector = "8aa3b61f"  # first 4 bytes of an app call's method-call arg, hex-encoded
lookup_url = f"https://scholtz.github.io/ARC56Registry/abi-signatures/{selector[:2]}/{selector}.txt"
try:
    signature = urllib.request.urlopen(lookup_url).read().decode().strip()
    print("Selector resolves to", signature)  # add(uint64,uint64)uint128
except urllib.error.HTTPError:
    print("No ABI method indexed for this selector yet")

Freshness

All three tables are regenerated daily by generate-hash-registry.yml and this site is redeployed right after by deploy-hash-registry-pages.yml. If two indexed specs compile to the same program bytes, the hash points at whichever spec file is larger (generally the more complete one) - see docs/hash-registry.md for the exact tie-breaking rule and the ABI signature registry's own rules.
